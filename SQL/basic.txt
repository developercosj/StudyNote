1. Row Chaining
- 데이터 크기가 커서 여러 블록에 나누어 저장하는 현상이며 검색시 2개 이상의 데이터 블록을 검색하기 때문에 성능 감소
- Initial Row Piece(행의 조각)와 Row Pointer 로 블록 내 저장된다.
- DB_BLOCK_SIZE 를 크게 하여 최소화가능하나 사이즈 변경이 어렵고 무조건 크게 할 수 없다.
- 데이터 저장시 발생

2. Row Migration
- update 로 행 길이가 증가하며 저장 공간이 초과일때 발생
- 원래 정보는 기존 블록에 저장하며 실제 데이터는 다른 블록에 저장한다.
- 검색 시 원래 블록의 주소를 먼저 읽고 다른 블록을 찾으며 성능이 감소한다.
- PCTEREE 영역을 충분히 할당 (너무 큰 할당 시 데이터 저장 공간 부족)
- update 시 발생

*Row Chaining 과 Row Migration 이 발생 시 많은 블록에 나누어서 데이터가 저장되므로 메모리, 디스크 IO(입력/출력)이 많이 발생하여 성능 저하
*컬럼의 수가 많은 테이블에서 데이터를 처리하게 되면 디스크 I/O 양이 증가하여 성능이 저하된다.
-> 코인거래소 근무시 코인의 수가 늘어날 수록 행이 늘어나는 것이 아니라 컬럼의 수가 늘어나게 짜여진 문제가 있었고 이것을 행을 늘리는 방식으로 바꾼 적이 있다.
코인의 수가 늘어날 수록 (상장시) 컬럼의 수가 증가하게 되어 성능저하가 발생했고 이 부분이 수정이 필수였음

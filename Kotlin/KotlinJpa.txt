- data class 로 선언하는 것보다는 일반 클래스로 선언하는 것이 더 좋다.
data class 는 open, abstract 키워드 사용지 못하기 떄문에 다른 클래스의 부모클래스가 될 수 없음

- 지연로딩 시 데이터 클래스에서 작동하지 않는 문제
- JPA 지향하는 Entity 객체는 Entity끼리도 상속성 이룰 수 있어야 한다.

# Entity
    - Entity는(클래스, 메서드, 인스턴스변수등 모두) final 클래스이면 안된다.
    - Entity는 최상위 클래스여야 한다.
    - Entity는 추상클래스, 구체 클래스 모두 될 수 있다.
    - Entity는 public 또는 protected 접근제한자를 가진 인수없는 생성자(no-arg)가 있어야 한다.
        (추가적으로 생성자를 정의 할 수도 있다.)
    - Enum 또는 interface 는 Entity가 될 수 없다.

# Entity 생성시 주의해야 할 점
    - 무분별한 mutable property 사용하지 말것
    - Kotlin 은 불변지향
    - Entity를 var 변수로 선언하고 외부에서 상태를 변경하도록 하지 말것 (캡슐화 필요) ㅌ


- 코틀린 클래스는 기본이 final (상속불가) 지연로딩을 위한 프록시 객체를 생성할 수 없기 때문에 클래스와 프로퍼티에 open 변경자 붙여줘야 한다.
- data class 를 사용하여 엔티티를 정의하게 되면 setter 의 개방을 막을 수 없음
    - 연관관계 매핑으로 인해 발생가능한 toString() 함수의 순환참조 문제 방지 못함
    - 동등한 엔티티에 대하여 예상치 못한 결과를 나타낼 수 있는 equals(), hashCode() 함수 문제

- var(variable)과 val(value)로 프로퍼티를 선언하며 var 로 선언하면 getter와 setter를 자동으로 생성, val 로 선언시 getter만 생성하고 불변


# Kotlin + JPA Lazy Loading 실무에서 많이 터지는 문제 5가지

1. Transaction 밖에서 Lazy 접근 (가장 많음)

- Lazy Loading 은 영속성 컨텍스트(Session) 필요
- Service 메서드 끝나면 Transaction 종료
- 그 후 Lazy 접근 -> 세션 없음
- LazyInitializationException -> Transaction 범위 or Fetch 전략 문제
- 해결 방법
    - Service 에서 DTO로 변환
    - Fetch Join 사용
    - 규칙 : Controller 에서 엔티티 Lazy 접근하면 사고남

2. OSIV (Open Session In View) 의존 문제

- OSIV 가 켜져 있으면 Controller 에서도 Lazy 가능하다. (spring.jpa.open-in-view=true)
- 문제점
    - Controller 에서 Lazy 호출 -> 쿼리 폭발
    - View 렌더링 중 DB 접근
    - 트랜잭션 경계 모호
    - 성능 예측 불가
- 실무 권장 : spring.jpa.open-in-view=false
- Service 에서 필요한 데이터 다 조회해서 DTO 로 반환


3. N + 1 문제 (성능 지뢰)

- 상황
val orders = orderRepository.findAll()

orders.forEach {
    println(it.member.name)  // Lazy
}

- 실행 쿼리
select * from order        -- 1번
select * from member where id=?  -- N번

- 해결 방법
1. Fetch Join
@Query("select o from Order o join fetch o.member")
fun findAllWithMember(): List<Order>

2. EntityGraph
@EntityGraph(attributePaths = ["member"])
fun findAll(): List<Order>


- 실무규칙
리스트 조회 + Lazy 접근 = 반드시 Fetch 전략 고민 필요


4. DTO 변환 시 Lazy 폭발
fun getOrders(): List<OrderDto> {
    return orderRepository.findAll().map {
        OrderDto(
            id = it.id,
            memberName = it.member.name
        )
    }
}

- map 내부에서 Lazy 접근
- 결과적으로 N+1 발생
- 해결
    - 처음부터 Fetch Join 사용하거나 또는 Query DTO 방식
